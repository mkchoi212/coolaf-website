The new kid in town: CARTHAGE

This is a tough one. My relationship with CocoaPods has always been a love and hate relationship. It allowed me to manage so many cool frameworks with such ease but at the same time, it was sometimes a pain in the ass. Especially with its unstable beta version that supported dependency management with Swift, our relationship was shaky at best

Podspec files and the cross-dependency issues made me want to cry but in the end, the relationship was one that I liked most of the time, so I kept at it. But one day, a new kid called Carthage showed up in town… At first, I didn’t take notice of it but after awhile, people started to notice how cool Carthage was so I said, “why not?”

CARTHAGE

Before one starts off with Carthage, one should know that folks at Github thought that while CocoaPods was generally easy to use, they wanted to something that was so easy that a little child could use it. That meant no podspec files or brand new .xcworkspaces being created.

How do I get it?

Go here and download the latest release and just follow the steps and you should be good to go!

Now that you have Carthage, there is literally only ONE thing you have to know. Remember Podfiles? That is now called Cartfile and it pretty much does the same thing as Podfiles.

To create one, simply go to the root directory of your project and make a file called “Cartfile”. Then in the file, add lines like this

github "Alamofire/Alamofire" == 2.0

A little bit about the syntax. == means what you think it means. Then there is >= 1.0 which means use version 1.0 or anything higher. Lastly, ~>1.0 means to use any version that is compatible with version 1.0.

Simple enough ,eh?

To install the new dependencies you’ve included in the Cartfile, you have to run the following command,

carthage update --platform iOS

which then shows stuff like this,

*** Fetching SwiftyJSON
*** Fetching Alamofire
*** Checking out Alamofire at "2.0.0"
*** Checking out SwiftyJSON at "2.3.0"
*** xcodebuild output can be found in /var/folders/h8/qp5163ln7mz2d7_l_yn6hd0m0000gp/T/carthage-xcodebuild.HBRNcq.log
*** Building scheme "Alamofire iOS" in Alamofire.xcworkspace

The whole process is like pod update or pod install!

Oh and by the way, that little platform option that we fed in is there to specify which platform Carthage will build the frameworks for. If you don’t specify that platform, Carthage will install to default platforms (usually both iOS and Mac).

Artifacts

Remember how CocoaPods made a special pods project that bound various frameworks with your Xcode workspace? Well, Carthage doesn’t do any of that junk and only builds the frameworks into the binary frameworks and it’s up to YOU to integrate it into the actually project.

After running,

carthage update

Carthage creates some files/folders for you.

You should Cartfile.resolved file because it shows the versions of the frameworks you’ve installed. And in the Carthage directory, you should see two folders. The first called Build contains the built frameworks for all of the dependencies. These are the ones you drag into your project. The Checkouts folder meanwhile is for Carthage’s smart way of managing the source code for all frameworks. It’s like caching the dependencies so the source doesn’t have to be copied over to other projects.

Also, remember that if you haven’t committed Build file, other people will need to run

carthage bootstarp

in order to actually compile the project.

The next part is where you actually drag the frameworks into your project. Here’s a little gif from Ray Wenderlinch that shows exactly how to do it. Video is worth probably million words from me.

Well, there you go. You learned something new today, right? Well I hope you did cause I know I did :) YAY

‘till next time.
